# HomeReminder — Harness Engineering 框架描述

> 基于 [Learn Harness Engineering](https://walkinglabs.github.io/learn-harness-engineering/) 课程内容构建。
> 命名规范：`.md` 文件大写，`.json` 文件小写。

---

## 1. 核心理念

Harness = **模型权重之外的一切工程基础设施**，由五个子系统组成：

| 子系统 | 职责 | 核心文件 |
|--------|------|----------|
| **指令** | agent 入口、约束、架构决策 | `harness/ARCHITECTURE.md`, `harness/CONSTRAINTS.md`, `harness/decisions/` |
| **工具** | agent 能调用什么命令 | `tools/init.sh`, `tools/verify.sh`, `tools/clean-state.sh` |
| **环境** | 依赖、运行时版本、可重现环境 | `tools/init.sh` + 项目配置文件 |
| **状态** | 跨会话进度、功能完成情况 | `harness/feature_list.json`, `harness/PROGRESS.md`, `harness/SESSION-HANDOFF.md` |
| **反馈** | 验证命令、评审评分 | `tools/verify.sh`, `harness/EVALUATOR-RUBRIC.md` |

### 各目录定位

| 目录 | 谁写 | 谁读 | 定位 |
|------|------|------|------|
| `harness/` | 所有 agent | 所有 agent + 人 | agent 工作状态（指令、进度、决策、交接） |
| `tools/` | 人 | 所有 agent | 全局工具脚本 |
| `agents/` | 人 | 对应 agent | 角色行为契约 |
| `templates/` | 人 | 所有 agent | 项目模板（work + src），不参与归档 |
| `work/` | agent | agent | 当前功能的跨 agent 产出 |
| `history/` | reviewer | 人 / agent | 已完成功能归档 |
| `docs/` | 人 | 人 | 人维护的架构文档、规范、设计说明 |

### 关键原则

- **仓库是唯一事实来源** — agent 看不到的东西对它来说就不存在
- **WIP=1** — 一次只一个功能处于 in_progress
- **三层验证** — L1 静态（lint/type check/架构边界规则检查）→ L2 运行时（`test/unit/` + `test/integration/`）→ L3 系统级（`test/e2e/`）
- **干活和检查分开** — planner/implementer 不评估自己
- **人审先行** — feature_list 填完后，你生成 overview.md 作为执行蓝图，审过再开始
- **模板驱动** — work 从 `templates/work/` 整体复制，src 模块从 `templates/src/` 复制

## 3. 整体流程

### 阶段 1：任务拆分（人审检查点）

```
你填 harness/feature_list.json
    │
    ▼
你读 feature_list.json → 生成 docs/task-breakdown/overview.md
    │
    ▼
审阅确认 → OK
    │
    ▼
planner 开始 WIP=1 循环
```

### 阶段 2：WIP=1 开发循环

#### 三角色

| 角色 | Skill | 职责 |
|------|-------|------|
| planner | `agents/planner/SKILL.md` | 选功能、标记 in_progress、出方案 |
| implementer | `agents/implementer/SKILL.md` | 从 `templates/src/` 创建模块 → 写代码 → 写测试 → 记录决策 |
| reviewer | `agents/reviewer/SKILL.md` | 三层递进验证、fix-queue、归档 |

#### 决策记录

implementer 开发过程中产生重大决策时，分级记录：

| 决策级别 | 写入位置 | 触发条件 |
|---------|---------|---------|
| **模块内** | `work/implementer/DECISIONS.md` | 模块内部选了算法 A 而非 B、数据结构的权衡、实现方式的取舍 |
| **项目级** | `harness/decisions/{feature-id}_{module-name}_{decision-summary}.md` | 跨模块处理决策、引入项目未使用的新技术/库/模式、变更全局接口约定 |

每条决策记录包含：**日期、上下文（为什么需要决策）、选项（考虑过哪些方案）、决策内容、影响范围**。

#### implementer 模块创建

implementer 开发新模块时，从 `templates/src/` 复制到 `src/<module-name>/`，同步在 `test/unit/` 下创建 `<module-name>/` 目录，然后补全 `ARCHITECTURE.md`（模块架构）、`CONSTRAINTS.md`（模块约束）、`PROGRESS.md`（模块进度，含测试覆盖率目标），并在 `code/` 中写入代码。

```
templates/src/                    src/
├── ARCHITECTURE.md               ├── auth/
├── CONSTRAINTS.md      cp -r     │   ├── ARCHITECTURE.md
├── PROGRESS.md         ──────→   │   ├── CONSTRAINTS.md
└── code/                         │   ├── PROGRESS.md
                                  │   └── code/
                                  │       └── auth_service.dart
                                  └── storage/
                                      ├── ARCHITECTURE.md
                                      ├── CONSTRAINTS.md
                                      ├── PROGRESS.md
                                      └── code/
                                          └── db.dart
```

#### 日志规范

**操作日志** — `work/logs/log.json`（小写），JSON 行追加：

```json
{"timestamp":"2026-05-27T14:30:00","agent":"planner","action":"select","feature":"F-03","detail":"标记 in_progress"}
{"timestamp":"2026-05-27T15:00:00","agent":"implementer","action":"create_module","feature":"F-03","module":"auth"}
{"timestamp":"2026-05-27T15:10:00","agent":"implementer","action":"decision","feature":"F-03","level":"project","detail":"引入 bcrypt 替代 argon2"}
{"timestamp":"2026-05-27T16:20:00","agent":"reviewer","action":"verify","feature":"F-03","layer":"L1","result":"pass"}
```

**测试日志** — `work/logs/tests/F{id}-L{layer}-round{n}.log`，验证命令的完整 stdout/stderr。

#### WIP=1 循环

```
planner 选 pending → 标记 in_progress → 追加 log.json
    │
    ▼
cp -r templates/work/ work/ → 补全 planner/BREAKDOWN.md + PLAN.md
    │
    ▼
implementer 读 PLAN.md → cp -r templates/src/ src/<module>/ → 写代码 → 补全模块 ARCHITECTURE/CONSTRAINTS/PROGRESS
    │  如有重大决策 → 写入 work/implementer/DECISIONS.md 或 harness/decisions/
    │  追加 log.json
    ▼
交给 reviewer
    │
    ▼
reviewer L1（静态分析：lint + type check + 架构边界规则检查）→ 补全 reviewer/L1-REPORT.md → 写 logs/tests/Fxx-L1-round{n}.log
    │ 不通过 → 补全 reviewer/FIX-QUEUE.md（每条包含"什么出错 + 为什么 + 怎么修"）→ 退 implementer
    │ 通过 → L2（`test/unit/` + `test/integration/` 全部通过）→ 补全 L2-REPORT.md → 写 tests/Fxx-L2-round{n}.log
    │   不通过 → 补全 FIX-QUEUE.md（每条失败附修复指引）→ 退 implementer
    │   通过 → L3（`test/e2e/` 端到端场景全部通过）→ 补全 L3-REPORT.md → 写 tests/Fxx-L3-round{n}.log
    │     不通过 → 补全 FIX-QUEUE.md（每条失败附修复指引）→ 退 implementer
    │     通过 → 补全 COMPLETION.md → 清空 FIX-QUEUE.md
    ▼
mv work/planner work/reviewer work/logs → history/<id>-<name>/
    │
    ▼
cp -r templates/work/ work/ → 重建
    │
    ▼
planner 更新 PROGRESS.md → 更新 feature_list.json（status → completed）
    │
    ▼
取下一个 pending → 循环
```

#### 关键规则

- **人审先行**：feature_list 填完后，你生成 overview.md 作为执行蓝图，审过再开始
- **模板驱动**：work 从 `templates/work/` 复制，src 模块从 `templates/src/` 复制
- **模板不归档**：`templates/` 永久保留，不参与归档
- **WIP=1**：同时只有一个功能 in_progress
- **标记先于执行**：planner 先改 status 再写方案
- **fix-queue 驱动修复**：implementer 每次启动先检查 FIX-QUEUE.md
- **每步必记日志**：写入 `log.json` 和 `tests/`
- **同功能退回 > 3 次**：记录到 PROGRESS.md 阻塞项，需人类介入

## 4. templates/work/ — work 模板

目录结构与 `work/` 一致，不参与归档。每个文件章节固定、内容留白（`<!-- xxx -->`）。

```
templates/work/
├── planner/
│   ├── BREAKDOWN.md
│   └── PLAN.md
├── implementer/
│   ├── DECISIONS.md
│   └── .gitkeep
├── reviewer/
│   ├── L1-REPORT.md
│   ├── L2-REPORT.md
│   ├── L3-REPORT.md
│   ├── FIX-QUEUE.md
│   └── COMPLETION.md
└── logs/
    ├── log.json                     # 操作日志模板（空 JSON 行）
    └── tests/
        └── .gitkeep
```

| 模板 | 使用方 | 核心章节 |
|------|--------|---------|
| `implementer/DECISIONS.md` | implementer | 模块内重大决策（时间倒序）：日期、上下文、选项、决策、影响 |
| `planner/BREAKDOWN.md` | planner | 功能 ID、工作单元列表、依赖拓扑、状态表、排除项 |
| `planner/PLAN.md` | planner | 涉及模块、实现步骤（验收标准）、依赖、排除项 |
| `reviewer/L1-REPORT.md` | reviewer | 验证命令、架构边界检查、模块约束检查、问题清单、建议固化 |
| `reviewer/L2-REPORT.md` | reviewer | 启动检查、排除项检查、测试结果、四维度评分、验收标准对照 |
| `reviewer/L3-REPORT.md` | reviewer | 端到端测试、用户场景模拟、资源清理、清洁状态 |
| `reviewer/FIX-QUEUE.md` | reviewer | 问题列表（验证层/评分维度/位置/实际/根因/期望/修复指引） |
| `reviewer/COMPLETION.md` | reviewer | 功能 ID、各层结果、四维度评分、模块质量、已知限制、遗留问题 |

## 5. templates/src/ — src 模块模板

每个 `src/` 子模块按此结构创建。implementer 执行 `cp -r templates/src/ src/<module-name>/` 后补全。

```
templates/src/
├── ARCHITECTURE.md     # 模块级架构
├── CONSTRAINTS.md      # 模块级约束
├── PROGRESS.md         # 模块级进度
└── code/               # 代码文件
    └── .gitkeep
```

| 文件 | 职责 |
|------|------|
| `ARCHITECTURE.md` | 模块概述、对外接口、内部结构、依赖 |
| `CONSTRAINTS.md` | 模块级硬约束（数据/接口/性能，用"必须/禁止"语言） |
| `PROGRESS.md` | 模块进度（工作单元 + 修复单元 + 阻塞项 + 测试覆盖率目标） |
| `code/` | 模块源代码 |

## 6. work/ — 当前功能 Agent 产出

| 文件 | 谁写 | 谁读 | 说明 |
|------|------|------|------|
| `implementer/DECISIONS.md` | implementer | reviewer, planner | 模块内重大决策 |
| `planner/BREAKDOWN.md` | planner | implementer, reviewer | 功能拆分 |
| `planner/PLAN.md` | planner | implementer | 实现方案 |
| `reviewer/L1-REPORT.md` | reviewer | implementer, planner | L1 静态分析结果 |
| `reviewer/L2-REPORT.md` | reviewer | implementer, planner | L2 运行时验证结果 + 四维度评分 |
| `reviewer/L3-REPORT.md` | reviewer | implementer, planner | L3 系统级确认结果 |
| `reviewer/FIX-QUEUE.md` | reviewer | implementer | 修复工作单（七字段含修复指引） |
| `reviewer/COMPLETION.md` | reviewer | planner, 人 | 验证完成、模块质量评分、已知限制 |
| `logs/log.json` | 三个 agent | 人, 所有 agent | 操作日志 |
| `logs/tests/F{id}-L{layer}-round{n}.log` | reviewer | 人, implementer | 测试输出 |

## 11. 使用指南

1. 确定技术栈 → 填 `harness/ARCHITECTURE.md` + `harness/CONSTRAINTS.md`
2. 在 `harness/feature_list.json` 中添功能
3. 写 `docs/architecture.md` + `docs/conventions.md`
4. 你生成 `docs/task-breakdown/overview.md` → 审阅确认
5. 审过 → WIP=1 循环（`cp -r templates/work/ work/` + `cp -r templates/src/ src/<module>/`）
6. 添加项目配置 → 修改 `tools/init.sh` + `tools/verify.sh`
