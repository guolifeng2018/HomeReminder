# 评审评分标准

---

## 评分等级

| 等级 | 含义 | 是否通过 |
|------|------|---------|
| A | 优秀，超出预期 | ✅ 通过 |
| B | 合格，满足验收标准 | ✅ 通过 |
| C | 有缺陷，需修复 | ❌ 不通过 |
| D | 严重缺陷或缺失 | ❌ 不通过 |

**规则**：任一维度 C 或 D → 整体不通过。

---

## 维度一：正确性

<!-- 代码行为是否符合验收标准 -->

| 等级 | 标准 |
|------|------|
| A | 全部验收标准通过，边界条件正确处理 |
| B | 全部验收标准通过 |
| C | 存在未通过的验收标准或已知错误 |
| D | 核心功能无法运行 |

---

## 维度二：架构合规

<!-- 是否遵守 ARCHITECTURE.md 和 CONSTRAINTS.md -->

| 等级 | 标准 |
|------|------|
| A | 完全合规，无任何违规 |
| B | 无硬约束违规，轻微风格偏离 |
| C | 存在硬约束违规 |
| D | 严重违反架构（如依赖方向颠倒） |

---

## 维度三：测试覆盖

<!-- 测试是否覆盖主流程和边界条件 -->

| 等级 | 标准 |
|------|------|
| A | 主流程 + 边界条件 + 异常路径均有测试 |
| B | 主流程全覆盖，边界条件有基本覆盖 |
| C | 仅覆盖主流程，缺少边界或异常测试 |
| D | 无测试或测试无法运行 |

---

## 维度四：代码质量

<!-- 仅做客观检查，不做主观评价 -->

| 等级 | 标准 |
|------|------|
| A | 命名清晰、结构合理、无重复代码 |
| B | 无明显问题 |
| C | 存在命名混乱、重复代码、过长函数 |
| D | 代码不可读或存在明显反模式 |
